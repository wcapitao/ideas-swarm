A Simple Batched Threshold Encryption Scheme
Guru-Vamsi Policharla
Commonware, Inc.
1
Introduction
In this note, we construct a simple batched threshold encryption scheme that satisfies censorship resistance,
does not suffer from epoch restrictions, and has quasi-linear decryption complexity in the batch size with a
CPA ciphertext size of |G1| + |GT |. Using standard techniques in the random oracle model, one can handle
arbitrary length messages with a ciphertext size of |G1| + |msg|. CCA security can be obtained by appending
a simulation-extractable NIZK proof of well-formedness of the ciphertext, which increases the ciphertext size
by 2|F|. We prove security under the Decisional Bilinear q-Power Diffie-Hellman Assumption [BGW05] in
asymmetric pairing groups, assuming the NIZK is zero-knowledge and straight-line simulation-extractable.
These improvements come at the cost of an interactive setup phase (involving secure multiplications) and
secret keys that grow linearly with the batch size. See Table 1 for a detailed comparison. For a discussion
on the relevance of this primitive to encrypted mempools and MEV, we refer the reader to [CGPP24] and
follow-up works (Section 4). We direct the reader to Section 5 for an overview of the construction.

Preliminaries
We use [n] to denote the set {1, 2, . . . , n}. The security parameter is denoted by λ ∈N. A function f : N →N
is said to be polynomial if there exists a constant c such that f(n) ≤nc for all n ∈N, and we write poly(·)
to denote such a function. A function f : N →[0, 1] is said to be negligible if for every c ∈N, there exists
N ∈N such that for all n > N, f(n) < n−c, and we write negl(·) to denote such a function. A probability
is noticeable if it is not negligible, and overwhelming if it is equal to 1 −negl(λ) for some negligible function
negl(λ). For a set S, we write s ←S to indicate that s is sampled uniformly at random from S. An algorithm
A is PPT (probabilistic polynomial-time) if its running time is bounded by some polynomial in the size of its
input.
Bilinear groups. Let GrpGen(1λ) output a description (G1, G2, GT , p, g1, g2, ◦) where G1, G2, GT are groups
of prime order p, g1, g2 are generators of G1, G2 respectively, and ◦: G1 × G2 →GT is an efficiently
computable bilinear map satisfying (i) bilinearity: [x]1◦[y]2 = [xy]T for all x, y ∈Zp, and (ii) non-degeneracy:
g1◦g2 ̸= 1GT . We write gT := g1◦g2 and, for b ∈{1, 2, T}, [x]b := x·gb. We work in the Type-III (asymmetric)
setting: no efficiently computable isomorphism between G1 and G2 is assumed.

Definitions
We recall the definitions of batched threshold encryption.
Definition 1 (Batched Threshold Encryption). A batched threshold encryption (BTE) scheme instantiated
with N ∈N parties, a threshold of t ∈[N], maximum batch size of B ∈N, and message space of M consists of
the following algorithms:
• Setup(1λ, 1B, 1N, 1t) →(ek, sk1, . . . , skN, dk): The setup algorithm takes the security parameter λ, the
maximum batch size B, the number of parties N, and the threshold t, and outputs a public encryption key
ek, a public decryption key dk, and N secret keys (sk1, . . . , skN).
• Enc(ek, m) →ct: The encryption algorithm takes the encryption key ek and a message m ∈M, and
outputs a ciphertext ct.
2

• PartialDec(skj, {cti}i∈[B]) →pdj: The (deterministic) partial-decryption algorithm takes a secret key skj
for a party j ∈[N] and B ciphertexts, and outputs a partial decryption share pdj or ⊥.
• Verify(dk, j, pdj, {cti}i∈[B]) →{0, 1}: The (deterministic) verification algorithm takes the decryption key
dk, a party index j ∈[N], a claimed partial decryption share pdj, and B ciphertexts, and outputs 1 iff pdj
is a valid share for party j on that batch.
• Combine(dk, T ⊆[N], {pdj}j∈T ) →pd: The (deterministic) combiner algorithm takes the decryption
key dk, a subset T ⊆[N], and the partial decryption shares of these |T| parties, and outputs a partial
decryption pd or ⊥.
• Dec(dk, pd, {cti}i∈[B]) →{mi}i∈[B]: The (deterministic) decryption algorithm takes the decryption key
dk, a partial decryption pd and B ciphertexts, and outputs the corresponding messages, where each mi ∈
M ∪{⊥}.
Let BTE = (Setup, Enc, PartialDec, Verify, Combine, Dec) be a BTE scheme.

Robustness.
A BTE scheme must be robust against malicious decryption servers or clients that submit
malformed ciphertexts or decryption queries. Informally, we require the following two properties:
• Consistency: It should be computationally infeasible for an adversary to produce two different sets
of partial decryptions for the same batch of ciphertexts such that both sets verify yet Combine yields
conflicting plaintexts.
• Rogue Ciphertext Security: It should be computationally infeasible for an adversary to craft a mal-
formed ciphertext that causes decryption of an honest party’s ciphertext in the same batch to fail.

Related Work
Batched Threshold Encryption (BTE) was introduced in [CGPP24]. While the initial work required an inter-
active setup protocol for each batch of ciphertexts decrypted, subsequent work [CGPW25, AFP25, SAS24,
BFOQ25] removed this but came with additional caveats. [CGPW25, AFP25, SAS24] showed how to build
a BTE scheme with a one-time setup but ciphertexts still had to be encrypted to a certain batch number
(epoch). Failure to be included in the chosen epoch would result in the ciphertext never being decrypted.
[FPTX25] partially addressed this by increasing the CRS size by a factor of k – which allows a user’s cipher-
text to be included in any of the k batches but at the end of k batches, the public key must be refreshed.
Furthermore, censorship resistance comes at the cost of a more expensive O(B log2 B) decryption complexity.
In a separate line of work, [BFOQ25] removes the epoch restriction but users must still choose a position
in the batch. This leaves transactions vulnerable to censorship: an attacker can outbid a victim for a conflict-
ing position and thereby block inclusion of the victim’s transaction. Follow-up work [ABD+25] improved the
decryption complexity to O(B log B) and also provided improvements over virtualization in the weighted
setting but still suffered from the same censorship issues.
4

Finally, [BNRT26] provided a construction based on partial fraction techniques which avoids both the
epoch restriction and censorship issues. However, this comes at the cost of an interactive setup (involving
secure inversions) and the size of secret keys held by each party grows with the batch size.
Also worth noting are two works [BCF+25] and [GWWW25] that build batched threshold encryption
and batched identity based encryption respectively, with silent setup.
Our work also requires an interactive setup phase (involving secure multiplications) and secret keys
held by each party grow with the batch size. It has the shortest ciphertext amongst all known schemes
|G1| + |GT |, supports quasi-linear decryption complexity O(B log B), and we believe it’s a conceptually
simpler construction.

Implementation and Benchmarks
To evaluate the concrete performance of our scheme, we implemented it in Rust, using arkworks [ac22] for
implementations of pairing-friendly curves. Our implementation can be found at https://github.com/
commonwarexyz/simple-bte.
Setup.
Experiments were run on an M5 MacBook Pro with 48 GB of RAM in single-threaded mode. We
use BLS12-381 [BLS03] as our pairing friendly curve. The compressed representation of G1, G2, and GT
elements is 48, 96, and 576 bytes respectively. Group exponentiations in G1 and G2 take approximately 68
µs and 193 µs respectively, and a pairing takes approximately 443 µs.

Benchmarks.
We now discuss microbenchmarks for various algorithms in our scheme.
• Encryption. Always takes approximately 0.47 ms irrespective of batch size and the number of parties.
• NIZK Verify. Verifies that a batch of ciphertexts is valid using batch verification of Schnorr proofs
[APB+04] which can be carried out using O(B) hashes and an MSM of size O(B).
• PartialDec. Computes the partial decryption share for a given party using an MSM of size O(B).
• Verify. Verifies a partial decryption share, by computing a multi-pairing of size O(B).
• Combine. Combines a set of partial decryption shares into a final decryption by computing Lagrange
coefficients and using an MSM of size O(N).
• Decryption. Decrypts a batch of ciphertexts using a G1-FFT of size O(B), a GT -FFT of size O(B), and
O(B) pairings.

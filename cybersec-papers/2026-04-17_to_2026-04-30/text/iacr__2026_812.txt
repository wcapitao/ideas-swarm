Abstract
Bitcoin’s scripting language cannot verify arbitrary computation natively, yet applications
such as trust-minimized bridges depend on this capability. Recent techniques employ garbled
circuits: the prover commits off chain to a garbled circuit encoding a verifier, designed so that
evaluating it on an invalid witness reveals a secret. Posting that secret on chain serves as a
fraud proof, allowing the verifier to claim the prover’s stake without any on-chain computation.
To evaluate the garbled circuit and recover the secret, the verifier needs the prover’s input
labels, which the prover must post on chain. Since Bitcoin charges permanently for block space,
minimizing this on-chain footprint is a primary design concern. Achieving malicious security
via cut-and-choose compounds this: the prover must produce multiple independently garbled
copies of the circuit, requiring one set of labels per copy.
We present Mosaic, a protocol that achieves malicious security via cut-and-choose but reduces
the on-chain footprint so that it is independent of the number of garbled copies.
The key
technique, first introduced by Eagen (Glock, 2025) in this setting, is polynomial label correlation:
labels across all N garbled copies are arranged as evaluations of a degree-t polynomial, so the
t shares revealed during cut-and-choose fall one short of the reconstruction threshold.
We
use adaptor signatures to arrange that the prover’s on-chain witness commitment reveals the
missing share as a byproduct; the evaluator then reconstructs labels for all unchallenged copies
by interpolation. We sketch why Mosaic is secure against a malicious prover and verifier and
instantiate it for trust-minimized Bitcoin bridging with a Groth16 verifier circuit, a full protocol
specification, and a Rust implementation.

Introduction
Bitcoin’s limited scripting language allows locking funds under simple conditions such as equality
checks, hash preimages and signatures. This expressiveness is enough to build payment channels
and time-locked contracts, but falls well short of verifying arbitrary computation. Bitcoin cannot
check whether a given input satisfies some predetermined NP relation.
This limitation becomes an obstacle in the following setting. Two parties, a prover P and a
verifier V , agree on an NP relation R and a public instance x. P claims to know a witness w such

that R(x, w) = 1, and V is willing to pay a prize in exchange for it. To deter P from submitting
an invalid witness, both parties agree that P puts up a stake, which V should be able to claim if
the witness is invalid. Since Bitcoin cannot evaluate R directly, the check must happen off chain,
but its outcome must be tied to on-chain consequences in a way that neither party can subvert.
Garbled circuits and on-chain input authentication.
Garbled circuits [18, 19] offer a natural
way to move the verification of R off chain. The key property we exploit is that we can arrange
for P to garble a circuit encoding the relation R and send it to V such that evaluation on an
invalid witness reveals a designated secret s∗, while evaluation on a valid witness reveals nothing.
Once both parties have agreed on the funds to be locked, they pre-sign a Witness Fault transaction
whose spending condition requires knowledge of s∗, so that V recovering s∗translates directly into
an on-chain claim against P’s stake.
What makes garbled circuits particularly well suited to this setting is that their input encoding
structure aligns naturally with Bitcoin’s transaction validation. The data P must publish on chain
to authorize her transaction can be arranged to simultaneously serve as the garbled circuit input
labels. There is therefore an exact correspondence between the witness P commits to on chain and
the input on which V evaluates the garbled circuit.
Malicious security and associated costs.
The above assumes P garbles honestly. A malicious
P could produce an incorrect garbled circuit that always evaluates to true, preventing V from
recovering s∗even for an invalid witness. The standard mitigation is cut-and-choose: P produces
N independently garbled copies of the circuit and commits to all of them before seeing which will
be checked. V randomly selects t of these to open and verify, leaving ℓ= N −t for evaluation, with
a statistical guarantee that at least one of the remaining copies is correctly garbled.
This introduces an on-chain cost problem. Authenticating P’s input to each of the ℓunchal-
lenged circuits requires posting the corresponding input labels on chain: one set of labels per
circuit. Bitcoin charges permanent block space for every byte in a transaction witness, which can
be prohibitive when the witness size is non-trivial.
Authenticated garbling (introduced in [20]) might appear to offer a cheaper alternative, but it
is incompatible with our setting’s timing constraints. The verification mechanism in authenticated
garbling operates during (or after) evaluation: the evaluator checks MACs on wire labels as the
circuit is computed, detecting a dishonest garbling only at that point. In our setting, evaluation
occurs only after both parties have locked their funds at Kickoff, so by the time authenticated
garbling could flag incorrect garbling, it is too late for V to decline participation. Cut-and-choose is
appropriate precisely because its challenge-and-open phase runs before any funds are committed: V
inspects the opened circuits and proceeds to lock funds only if they are correctly garbled, obtaining
a statistical guarantee over the remaining copies.
Our contribution.
We present Mosaic, a protocol that achieves malicious security while reducing
the on-chain cost to a single set of labels per input wire, regardless of ℓ.
The key idea is to correlate the labels across all N garbled circuits using a threshold secret-
sharing scheme: the labels for a given wire and input value across all N copies are arranged as
evaluations of a polynomial, with reconstruction threshold t + 1.
The t labels revealed during
cut-and-choose are t shares of this polynomial, one short of the reconstruction threshold. Posting
a single additional label on chain per wire, the one that completes the threshold, allows V to
interpolate the polynomial and derive the correct labels for all ℓunchallenged circuits algebraically.
The on-chain footprint is therefore independent of ℓ.

We informally argue why Mosaic is secure against a malicious prover and a malicious verifier,

Relation to prior work.
The idea of using garbled circuits for optimistic verification on Bitcoin
originates with Rubin [7] and Eagen [6] who independently observed that if a garbled circuit encodes
a SNARK verifier, evaluation on an invalid proof reveals a secret that can serve as a fraud proof,
replacing the bisection-based fraud proofs of BitVM [8, 9] with simple secret revelation.
Eagen [6] presented this observation as the Garbled Lock (Glock) primitive and proposed
Glock25, the first concretely practical instantiation that achieved malicious security. Glock25 intro-
duces several ideas that Mosaic builds upon: (i) using Feldman verifiable secret sharing to correlate
labels across cut-and-choose copies, so that opening a challenged subset during setup leaves the
shared secret information-theoretically hidden; (ii) using Schnorr adaptor signatures, rather than
Lamport signatures, as the on-chain authentication mechanism, so that completing a signature si-
multaneously authorizes the Bitcoin transaction and reveals a polynomial share; and (iii) combining
these techniques so that the on-chain footprint per input wire is a single Schnorr signature, inde-
pendent of the number of unchallenged circuits. The use of secret sharing within cut-and-choose
itself has roots in the secure two-party computation literature: Lindell and Pinkas [10] applied it to
output-wire determination, and Huang, Katz, and Evans [11] used Feldman VSS on output labels
in their symmetric cut-and-choose protocol.
Mosaic provides a complete, self-contained treatment of this approach. Our contributions rela-
tive to prior work are as follows. We give a full formal construction (Section 6) with explicit protocol
steps, a concrete Bitcoin transaction graph (Section 2), and detailed cost analysis (Section 9) backed
by an open-source Rust implementation [26], whereas Glock25 describes the mechanism at a higher
level of abstraction. We formally state the security properties required in our setting and sketch
how they follow from the underlying cryptographic primitives and the Bitcoin consensus protocol.
We also introduce a three-way partition of circuit inputs that reflects the distinct timing at which
different parts of the statement become known in a Bitcoin protocol. Each phase maps to a different
adaptor signature mechanism with different costs.
Within the broader BitVM landscape, Chen [12] provides a systematization of knowledge orga-
nizing the design space. Recent works [14, 13, 15] significantly reduce the size of garbled tables for
verifying Groth16 proofs and we explore in Section 9 how Mosaic immediately benefits from some
of these improvements.

Fair Exchange on Bitcoin
Before presenting the Mosaic cryptographic protocol, we describe the fair exchange protocol it is
designed to support and formalize the properties it must achieve.
Earlier works like [27] study fair exchange on Bitcoin, but rely on the predicate check being
expressible in Bitcoin script and compatible with Bitcoin’s transaction-size, script-validation, and
standardness limits.
Recall we are in the setting where a prover P and a verifier V agree on an NP relation R, a
public instance x, a prize pB, and a stake sB. P claims to know a witness w with R(x, w) = 1.
We wish to design a fair exchange protocol where P receives the prize if her witness is valid, and
forfeits the stake to V if it is not, with no trusted third party.

Transaction graph for fair exchange.
Bitcoin transactions operate in the UTXO (unspent
transaction output) model. A transaction tx consumes one or more existing UTXOs as inputs
and produces one or more new UTXOs as outputs. Each output carries a value and a spending
condition: a script that specifies who may spend the output and under what circumstances. A
transaction is valid only if, for every input it consumes, the accompanying witness data satisfies
the corresponding spending condition. A UTXO can be spent at most once; if two transactions
attempt to spend the same UTXO, consensus ensures at most one is confirmed.
The parties encode the fair exchange contract as a directed graph of transactions, each with
spending conditions that capture the different outcomes. To enforce the logic of dispute resolution
and timeout paths so it cannot be unilaterally altered after setup, certain transactions are made
non-malleable. In practice, this is done by having such transactions collaboratively pre-signed by
both users.
The key cryptographic object is a fault secret s∗, committed to by P during setup. The spending
condition of the Witness Fault transaction requires knowledge of s∗. Mosaic guarantees that V
learns s∗if and only if P’s on-chain committed witness fails R, so this transaction can be posted
by V exactly when P has cheated, awarding all funds to V .

The graph in Figure 2 has the following transactions.
• Kickoff. Parties lock their funds here: P contributes the stake s and V contributes the prize
p. Before the Kickoff transaction is confirmed on chain, either party controls their UTXO
and can back out unilaterally. The exact mechanism for co-signing is described in Section 9.
• Witness. Broadcast by P to commit to her witness on chain. The spending condition of the
Kickoff output requires P to post one signature per byte of w under V ’s key. Because P does
not hold V ’s signing key, the only way she can produce these signatures is by completing pre-
signatures that V provides before the Kickoff transaction is broadcast, one per possible byte
value per position in w. Each pre-signature is constructed so that completing it simultaneously
produces a valid Bitcoin signature and reveals the corresponding garbled circuit input label
to V . This is possible because Mosaic is designed such that on-chain witness commitment
maps precisely to garbled circuit input labels.
• Witness Fault. Spendable by V using s∗. If the Witness transaction encodes a failing input,
Mosaic guarantees V recovers s∗by evaluating the garbled circuit on the revealed input labels.
Posting this awards both funds to V and protects an honest V against a malicious P.
• Witness Timeout. Spendable by V after a fixed time window ∆Witness if P has not broad-
cast the Witness transaction. This forces P to commit to a witness within the allotted period.
• Payout. Spendable by P after a delay ∆Payout, provided neither Witness Fault nor Witness
Timeout has been posted. If P posts a valid witness that V cannot fault, P collects both
funds once the delay expires. This protects an honest P against a malicious V .
• Cooperative Payout. Spendable immediately by P with a co-signature from V . During
the ∆Witness window, P can send V the witness off chain; if V accepts, both parties settle
immediately without posting the Witness transaction at all.

Input timing.
Not all input to the garbled circuit is known at the same point in the protocol.
Some part of the public statement x is fixed before any funds are locked. However, there may also
be public data that depends on the Kickoff transaction’s identifier and is only determined once the
UTXOs that fund this transaction have been agreed upon by both parties. The witness itself is
committed last, via the Witness transaction. Mosaic partitions the circuit inputs into three types
accordingly: Phase 1 inputs x1 ∈{0, 1}n1, known from the start; Phase 2 inputs x2 ∈{0, 1}n2,
fixed once the Kickoff transaction is confirmed; and Phase 3 inputs x3 ∈{0, 1}n3, corresponding to
the witness and committed on chain last.
2.1
Properties of Fair Exchange
We formalize the properties that the fair exchange protocol of Figure 1 must satisfy. Throughout,
we model time in discrete rounds (block heights) and write τ0 for the round at which the Kickoff
transaction confirms. The protocol is parameterized by two relative timelocks: ∆Witness (the window
for P to post the Witness transaction) and ∆Payout (the delay before P may claim the Payout).
We assume the Bitcoin consensus protocol operates correctly, confirming only valid transactions
whose witness data satisfies the corresponding spending conditions. In particular, we rely on three
properties:
• Persistence. A transaction confirmed at some depth k (included in a block with k−1 successor
blocks built on top of it) remains confirmed in all future honest views.
• Liveness. A valid transaction broadcast by an honest party is confirmed within ∆conf blocks.
• Double-spending protection. If two transactions spend the same UTXO, at most one is con-
firmed. This ensures the branches of the transaction graph are mutually exclusive.
Protocol properties.
The fair exchange protocol satisfies the following three properties.
• Completeness. If both P and V are honest and P holds a valid witness w with R(x, w) = 1,
then the protocol terminates with P receiving the stake and prize and V learning w.
• Fairness with Compensation. The protocol ensures that an honest party is never disad-
vantaged by a malicious counterparty:
(a) Prover security. For every PPT adversary A controlling V : if P is honest and holds a
valid witness w with R(x, w) = 1, then after the protocol terminates, P receives at least
the stake and prize, except with negligible probability.
(b) Verifier security. For every PPT adversary A controlling P: after the protocol termi-
nates, exactly one of the following holds except with probability at most 1/

Notation
Let G be a prime-order group of order p with generator G and scalar field Fp. We write [s] := s · G
for point-scalar multiplication, which serves as a commitment to the scalar s.
Parties.
We write G for the Garbler and E for the Evaluator. In the fair exchange setting of
Section 2, the prover P takes the role of G, and the verifier V takes the role of E. We use P and
V when discussing the fair exchange protocol and its properties, and G and E when discussing the
cryptographic protocol and its construction.
Parameters.
• κ: computational security parameter.
• n = n1 +n2 +n3: total number of input wires, partitioned into Phase 1, Phase 2, and Phase 3
wires. Each wire carries a byte value v ∈{0, . . . , 255}. 1.
• N: number of garbled circuits.
• t: number of challenged circuits; equivalently, the degree of the underlying polynomials.
• ℓ:= N −t: number of unchallenged circuits.
• C: a public Boolean circuit encoding the NP relation R, with n byte-valued input wires and
a single-bit output.
Notation conventions.
We index wires by j ∈[n] and byte values by v ∈{0, . . . , 255}. Circuit
indices run from 1 to N; index 0 is reserved for the secret in the secret-sharing scheme. Wires
1, . . . , n1 are Phase 1 wires; n1 +1, . . . , n1 +n2 are Phase 2 wires; and n1 +n2 +1, . . . , n are Phase 3
wires. We write C ⊂[N] for the challenge set of size t and U := [N] \ C for the unchallenged set of
size ℓ.

Security properties.
We rely on two standard properties of Feldman VSS:
1. Secrecy.
For an honest dealer, the commitment Cf together with any set of at most t
shares is information-theoretically independent of f(0). In particular, t shares of a degree-t
polynomial are consistent with every possible value of f(0).
2. Binding. Under the discrete logarithm assumption, it is infeasible to produce a commitment
Cf and two sets of t + 1 valid shares that interpolate to different secrets.
The Feldman
commitment therefore binds the dealer to a unique polynomial and a unique secret f(0).

3. Unforgeability (aEUF-CMA). An adversary with access to signing and adaptor generation
oracles cannot produce a valid signature on a fresh message.

Relation to standard definitions.
The standard garbling scheme formalism [17] introduces the
following security properties: privacy, obliviousness, and authenticity, of which we require only the
last. We tailor the definitions to the fraud-proof setting in several ways. In our setting, we collapse
the formalism to three algorithms: input labels are external parameters to GenTable (absorbing
an encoding procedure) and the output is a single field element compared against a commitment
(eliminating the need for a decoding procedure). CommitTable provides binding commitments for
cut-and-choose, and generation is deterministic from a seed so that challengers can re-derive tables
without receiving them.
We also emphasize that, in our setting, we require correctness holds
for failing inputs (C(x) = false yields sout) and authenticity protects sout from satisfying inputs,
reflecting that the secret is evidence of fraud rather than a proof of knowledge. Finally, our wires
carry byte values (256 labels per wire) rather than single bits.

Construction Overview
Before formally presenting Mosaic, we provide an overview of how the building blocks of Section 4
compose to achieve the properties required in Section 2.
Polynomial label correlation.
A central design choice in Mosaic is that the N garbled circuits
are not generated independently. For each input wire j ∈[n] and byte value v ∈{0, . . . , 255}, the
Garbler samples a uniformly random degree-t polynomial fj,v ∈Fp[X]≤t and uses fj,v(i) as the
input label for wire j, value v, in circuit i. A single output polynomial fout of degree t is sampled
similarly, and fout(i) is embedded in circuit i as the false output label. The fault secret is the
constant term:
s∗:= fout(0).
G publishes Feldman commitments Cj,v = Com(fj,v) and Cout = Com(fout) for all j, v, along with
commitments to each of the N garbling tables. These commitments bind G to a unique polynomial
per wire-value pair and to a unique s∗, before any challenge is issued.
Cut and choose.
E challenges a uniformly random subset C ⊂[N] of size t. For each challenged
circuit i ∈C, G opens the garbling seed, all input labels {fj,v(i)}j,v, and the output share fout(i).
E verifies each opening against the Feldman commitments and re-derives the garbling table from

7.2
Safety
Claim 2. If G honestly executes the protocol with an input satisfying C(x1, x2, x3) = true, then no
PPT adversary playing the role of E can recover s∗with non-negligible probability.
Proof sketch. If an adversarial E provides maliciously generated adaptors at step D3., G aborts at
step D4.. Since the Kickoff transaction has not yet been broadcast, no funds are locked and G
suffers no loss. We consider the case where adaptor verification passes and the protocol reaches the
Witness Resolution phase.
The adversary’s view consists of the Feldman commitments, t opened input and output polyno-
mial shares and garbled tables from the challenge set, unchallenged garbling tables, and completed
signatures on the Witness transaction. We argue that none of these reveal s∗.
The t output shares {fout(i)}i∈C are evaluations of a degree-t polynomial at t non-zero points.
By the secrecy of Feldman VSS, these are information-theoretically independent of the constant
term s∗= fout(0).
For each unchallenged circuit i ∈U, the garbling table Ti was generated with fout(i) as the
false-output label. Since C(x) = true, holding appropriate input labels and an honestly generated
garbling table Ti does not help an adversary recover fout(i) by authenticity of the garbling scheme
(Definition 4). The adversary also sees [fout(i)] via the public commitment Cout, but this does not
reveal fout(i) by the hiding property of the commitment. Similarly, it is infeasible to recover s∗
directly from its public commitment Y ∗= [s∗].
The completed Witness transaction signatures reveal the reserved input shares {fj,v∗
j (0)}j, which
are evaluations of polynomials sampled independently of fout. G’s adaptor verification at step D4.
ensures the adversary cannot engineer adaptors whose completions would reveal information about
fout.

Instantiation on Bitcoin
In this section, we instantiate Mosaic in the setting of a trust-minimized Bitcoin bridge and explain
how the abstract roles and protocol phases of Section 6 map to a concrete deployment. We consider
a bridge in which a federation of operators controls a Bitcoin multi-signature wallet backing assets
on a sidechain. To participate in the bridge, each operator locks a stake that can be slashed if the
operator makes a dishonest withdrawal claim. Users lock Bitcoin into the bridge to mint sidechain
tokens and later redeem those tokens by requesting a withdrawal on Bitcoin.
To service a withdrawal, an operator first fronts the corresponding Bitcoin payment to the user
and then seeks reimbursement from the bridge by submitting a proof that the withdrawal was
19

fulfilled correctly. Mosaic is not used to prove this claim directly. Rather, it is used to enable a
counterproof: if the claiming operator submits an invalid proof, another operator can demonstrate
that the proof is false for the particular withdrawal instance under dispute.
Concretely, each withdrawal claim is tied to a specific deposit index, and the identity of the
claiming operator is fixed during an initial operator setup. Together, these values determine the
statement against which the claim is checked. Any non-participating operator can act as a watch-
tower. If the claimant submits an invalid proof, a watchtower can challenge by showing that the
submitted proof does not verify for that operator key and deposit index. Mosaic guarantees that
a valid counterproof causes the dishonest operator’s stake to be slashed, while an honest operator
cannot be punished.
9.1
Cryptographic Instantiation
This instantiation is chosen to align all cryptographic objects with mechanisms already native to
Bitcoin.
Group.
All group operations use secp256k1, the elliptic curve that Bitcoin uses natively. The
scalar field Fp is the secp256k1 scalar field, and G is the standard generator. This fixes the group
for all Feldman commitments, for the [s] := s · G commitment notation used throughout, and for
the adaptor signatures.
Schnorr adaptor signatures.
We instantiate the adaptor signature scheme of Section 4.2 using
BIP-340 Schnorr signatures over secp256k1 [21]. The hash function H in GenAdaptor is the BIP-340
tagged hash. Accordingly, the Witness transaction spending condition can be expressed using a
standard BIP-340 signature check per wire, without any custom verification logic.
As observed by Linus [5], a single public key suffices for all n2 + n3 signatures while binding
each to a distinct message; the exact script is given in [5].
Verifiable secret sharing.
We use Feldman VSS over secp256k1 as defined in Section 4.1.
Garbling scheme.
We use a privacy-free garbled circuit scheme, one concrete instantiation of
which is the half-gates construction [22] adapted to the privacy-free setting.
We also estimate
costs based on promising recent work [13, 14, 15] that significantly reduces garbled table size for a
Groth16 verifier. Labels are elements of {0, 1}κ obtained by truncating the binary representation
of field elements fj,v(i) ∈Fp.
9.2
The NP Relation
A primary motivation for Mosaic is the problem of trust-minimized bridges for Bitcoin. The central
challenge is that Bitcoin’s scripting language cannot verify arbitrary computation on chain and in
particular, it cannot directly verify a SNARK proof. The BitVM line of work [8, 9] addresses this via
an optimistic protocol: a prover commits to a garbled SNARK verifier off chain, and if the prover
later submits an invalid proof, the verifier uses the garbled circuit to extract a fault secret and post
a fraud proof on chain. Indeed, the fair exchange protocol of Section 2 is a simplified instance of this
setting that isolates the core properties Mosaic must provide; in the bridge application, the same
properties are required to ensure that a watchtower can provide a valid counterproof in exchange
for slashing the operator of their stake. The NP relation is a Groth16 verifier: R(x, w) = 1 iff w
20

is a valid Groth16 proof for the statement x. The circuit C encoding R takes three types of input
corresponding to the three wire partitions.
• Phase 1 wires (n1 = 32): the operator public key, agreed upon before any funds are locked
and fixed for the lifetime of the contract. This is used to ensure that the proof being challenged
is authenticated by the operator.
• Phase 2 wires (n2 = 4): the deposit index, which identifies the specific instance of the
relation being proved. This is determined by the Kickoff transaction’s identifier and so cannot
be fixed until the Adaptor Setup phase. This ensures that proofs are tied to the particular
instance and cannot be reused across independent instances of deposits and withdrawals.
• Phase 3 wires (n3 = 128): the Groth16 proof bytes, which constitute the witness w that
watchtower commits to on chain via the Witness transaction.
The circuit C verifies that the 128-byte Groth16 proof submitted by the operator for a particular
operator key and deposit index is invalid.
9.3
Protocol Timeline
The three phases of Mosaic map onto the Bitcoin timeline as follows.
• Garbling Setup takes place entirely before the Kickoff transaction is broadcast. The gar-
bling and cut-and-choose interaction depend only on the agreed circuit C and the setup inputs
x1; neither the transaction graph nor the Kickoff txid is needed. Neither party has committed
any funds and either can walk away without loss.
• Adaptor Setup begins once both parties have agreed on the input UTXOs that will fund
the Kickoff transaction, since the Kickoff txid is needed to construct the messages {mj}
that the adaptors are bound to.
Both parties first pre-sign the Witness Fault, Witness
Timeout, Payout, and Cooperative Payout transactions.
The Adaptor Setup phase then
runs: V generates and sends the adaptors, P verifies them, and both parties co-sign the
Kickoff transaction. P broadcasts the Kickoff transaction; once it confirms, the timelocks
∆Witness and ∆Payout begin.
• Witness Resolution corresponds to P broadcasting the Witness transaction before ∆Witness
expires. The Witness transaction’s input contains the completed adaptor signatures (steps
W1.–W3.), one per Phase 2 and Phase 3 wire, which simultaneously authorize the transaction
on chain and deliver the garbled circuit input labels to V . V evaluates the garbled circuit
off chain (steps W4.–W7.): if P’s witness fails R, V recovers s∗and broadcasts the Witness
Fault transaction, claiming both funds.
Outside the three protocol phases, two additional paths may be taken depending on the behavior
of the parties. During the ∆Witness window, P may send V her witness off chain; if V accepts, he
co-signs the Cooperative Payout transaction and both parties settle without posting the Witness
transaction at all. This cooperative path is the expected execution when both parties are honest. If
instead P does not post the Witness transaction before ∆Witness expires, V broadcasts the Witness
Timeout transaction and recovers both funds; if P posts the Witness transaction but V does not
post Witness Fault before ∆Payout expires, P broadcasts the Payout transaction and recovers both
funds. Before the Kickoff transaction confirms, either party can abort at any time by spending
their input UTXO elsewhere.

9.4.2
Concrete Instantiation with PFHG
Using parameters n1 = 32, n2 = 4, n3 = 128 (n = 164), N = 181, t = 174, ℓ= 7, and the state-
of-the-art Groth16 verifier Boolean circuit [23] with 3 billion AND gates, the PFHG garbling table
size is |T| = 3×109 ×κ = 48 GB. Our implementation uses g16 [24], a fork of the BitVM Alliance’s
garbled SNARK verifier [23] modified for compatibility with SP1 proof structure. Garbling and
evaluation are performed using ckt [25], our privacy-free garbled circuit implementation optimized
for circuits at the billion-gate scale. To the best of our knowledge, before ckt, no general-purpose
garbling library could handle circuits of this size within practical memory bounds.
The garbling tables dominate communication and storage. Setting ℓsmall and t large (many
garblings need to be performed, but only few garbled tables need to be transmitted) is appropriate
in this regime since garbling with PFHG involves only symmetric-key operations, keeping per-table
compute low relative to table size.
The on-chain footprint is independent of ℓand |T|, following directly from the polynomial label
correlation. The Witness transaction comprises around 100 bytes of non-witness data (version,
input, output, etc) and approximately 9.0 kB of witness data (signatures, locking script, and
control block). The total raw transaction size is approximately 9.1 kB; however, under SegWit’s
weight rules, witness bytes receive a 75% discount, yielding an effective size of approximately 2.4
kvB. Both figures are well within Bitcoin’s standardness limits.
We provide an open-source production-grade Rust implementation of Mosaic [26]. Based on
single-threaded benchmarks on Apple M1 Pro, Garbling Setup takes approximately 6.3 hours for
G and 5.9 hours for E, Adaptor Setup takes approximately 2 seconds for G and 1 second for E, and
Witness Resolution is negligible for G with E requiring between 3 and 21 minutes (6 minutes in
expectation).
